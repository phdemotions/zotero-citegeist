## The one-line truth
> The identifier fallback chain works invisibly for the common case — but when it can't find data, the UI says nothing useful about *why* or *what to do next*, leaving a researcher who saved a conference paper from PubMed staring at a blank pane and a message that implies their item simply has no identifier.

## 🟠 Should fix

**1. The "No identifier" empty state in the pane doesn't tell you what identifiers are accepted — and one of them now requires manual entry.**
`src/modules/citationPane.ts:209`

You click a conference paper in your library. The pane shows: "No DOI, PubMed ID, or arXiv ID found for this item." Full stop. You don't know whether Citegeist looked in the Extra field, the Archive ID field, or the URL. You don't know what format to use. A researcher who just came back from a PubMed import that happened to strip PMIDs has no idea that adding `PMID: 12345678` to the Extra field would make Citegeist work.

The README FAQ documents the format correctly. The pane — the only surface a researcher sees when they're staring at a blank — does not. This is the highest-friction moment in the entire new feature: the researcher whose item *could* be covered but isn't, who could fix it themselves with one piece of information they aren't given.

Fix: expand the empty state to include a one-liner that points to the action: "No DOI, PMID, arXiv ID, or ISBN found. Add `PMID: 12345678` to the Extra field to enable lookup." Or use two lines — the identifier list, then the actionable hint. Either is better than the current dead-end.

---

**2. The right-click menu batch-fetch and the "View Citing Works / View References" items silently ignore non-DOI identifiers.**
`src/modules/menu.ts:48`, `src/modules/menu.ts:101`

The batch-fetch filter (`i.getField("DOI")?.trim()`) and the visibility guard for "View Citing / View References" (`!!items[0].getField("DOI")?.trim()`) both check only the DOI. An item that Citegeist successfully resolves via PMID or arXiv will:

- Be skipped by "Fetch All Citation Counts" on a collection
- Have "View Citing Works" and "View References" hidden in its right-click menu, even after citation data has been fetched and cached

The pane correctly uses `extractIdentifier(item)`, which checks the full chain. The menus were not updated to match. This creates a direct contradiction: a researcher with an arXiv preprint in their library sees citation data in the pane but can't reach the citation network from the right-click menu. They will try right-clicking, see no option, and conclude the feature is broken.

Fix: replace the DOI guards in `menu.ts` with `extractIdentifier(item) !== null` (importing `extractIdentifier` from `citationService`) for the batch-fetch eligibility check, and the same for the `popupshowing` visibility guard — noting that `showCitationNetwork` internally calls the service which will resolve via the identifier chain.

## 🟡 Consider

- **The book zero-suppression message has two different surface-level explanations that say different things.** In the pane (`citationPane.ts:351`), the message is "Citation tracking for books is limited in OpenAlex." In the pane collapse header (`citationPane.ts:35`), it's "Found on OpenAlex." These describe the same state from different angles, which is fine — but a researcher who opens the pane after seeing "Found on OpenAlex" in the header and then reads "Citation tracking for books is limited" may not connect them as the same item. It's a minor seam, not a broken experience. Consider aligning the framing: either both lead with the positive ("found") or both lead with the caveat ("limited").

- **The section summary for the "no identifier" state says "No identifier" — a technical label, not a researcher-facing description.** `citationPane.ts:210` sets `setSectionSummary("No identifier")`. This text appears in the Zotero pane header and in pane summaries. "No identifier" is developer vocabulary. "Not tracked" or "No lookup ID" communicates the same state without implying an error the researcher may not understand. Low-stakes, but it contributes to the "feels like a debug message" impression.

- **ISBN is not mentioned in the pane empty state.** The no-identifier message in the pane reads "No DOI, PubMed ID, or arXiv ID found for this item" but does not mention ISBN. A researcher with a book that has no ISBN in Zotero (common for older books imported from library catalogs) will see this message and not know that adding an ISBN would help. The README covers this; the pane does not.

## ⚪ Delights missed

- **The fallback chain runs silently — a researcher never knows which identifier Citegeist used.** This is intentional and usually right. But a power user who wants to know whether their arXiv preprint resolved via the Extra field, the Archive ID, or the URL has no way to see that. A tooltip on the pane's citation count, or a single line in the trend section ("Resolved via arXiv ID"), would be genuinely appreciated by the researchers who care. This is a fifth-session delight, not a blocker.

- **The book-suppression note ("Citation tracking for books is limited in OpenAlex.") disappears entirely when the citation count is non-zero.** That's correct. But there's a missed opportunity: for a book that *does* have citations, a parenthetical like "(book coverage may be incomplete)" alongside the count would help a researcher understand why their well-cited book shows only 4 citations. This is speculative — don't add it unless validated by user feedback.

## 🏆 If you fix one thing

**Update the menu.ts DOI guards to use `extractIdentifier(item) !== null`** — because this is the one change that makes the PMID/arXiv/ISBN chain complete end-to-end. Right now the pane tells you about citations it found via PMID, but right-clicking the same item hides the network browser. That contradiction will erode trust in the feature faster than any empty-state wording.
