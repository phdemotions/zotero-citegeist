## 🔴 Blockers

1. **bumpp v9.11.1 → v10.4.1 — four high-severity CVEs in transitive `tar` dependency**
   - Affected advisories: GHSA-34x7-hfp2-rc4v, GHSA-8qq5-rm4j-mr97, GHSA-83g3-92jg-28cx, GHSA-qffp-2rhf-9h96
   - Nature: path traversal, symlink poisoning, arbitrary file create/overwrite via hardlink
   - These fire during `npm run release`, which invokes bumpp and pulls the vulnerable `tar` through `giget → c12 → bumpp`
   - Fix: `npm install bumpp@^10` — breaking change is CJS build dropped (ESM only). The project uses `import` throughout; `npm run release` script calls bumpp as a CLI, not via `require()`, so no code changes expected
   - `package.json` range `^9.9.0` must become `^10.4.1` or `^10`

## 🟡 Consider

- **TypeScript 5.9.3 → 6.0.2 — major version, significant breaking changes**
  - TS 6.0 changes strict defaults (already on), drops `target: es5`, removes AMD/UMD, `--baseUrl`, `--outFile`, `--moduleResolution node`, and changes `types` field to empty-by-default. The `tsconfig.json` here targets Firefox 115 via esbuild, not tsc emit, so most removed emit options are irrelevant — but the `types` field change and strict-default shifts could cause type-check regressions
  - Not urgent; wait for ecosystem tooling (e.g., `@typescript-eslint` v8 support for TS 6) to stabilize before upgrading
  - The current `^5.7.0` range correctly excludes TS 6

- **globals 16.5.0 → 17.4.0 — minor version bump**
  - Used in `eslint.config.mjs` for browser/node global definitions. Minor bumps here are usually additive. Low risk, low urgency — can batch with the next eslint update cycle
