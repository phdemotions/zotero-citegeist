## The one-line truth
> The ISBN identifier was added silently — every user with a book in their library who has no DOI will now be told "No DOI, PubMed ID, or arXiv ID found" when the correct message should mention ISBN too, and this is a lie the product tells about its own capabilities.

---

## 🔴 Blockers

**1. Stale "no identifier" copy in the pane — `src/modules/citationPane.ts:209`**

The pane's fallback message was not updated when ISBN support was added to `extractIdentifier`. A researcher with a book-only entry (no DOI, no PMID, no arXiv) will see:

> "No DOI, PubMed ID, or arXiv ID found for this item."

But `extractIdentifier` now also tries ISBN (step 6 in `citationService.ts`). If the item has an ISBN, the pane will actually fetch correctly — so this message only fires when *all four* identifier types are absent. That means the copy actively misleads users: it implies trying to add an ISBN won't help, when it would.

**Fix:** Update to: `"No recognized identifier found. Add a DOI, PMID, arXiv ID, or ISBN to enable citation data."`

This is a Blocker because it ships an actively wrong instruction about the product's capabilities to a user audience (book-heavy researchers) that v1.1.0 was explicitly designed to serve better.

## 🟠 Should fix

**1. "Found on OpenAlex" empty-state in the section header is too bare — `src/modules/citationPane.ts:35`**

When a book has zero citations, the collapse-header summary reads "Found on OpenAlex." This is factually correct but communicatively thin. A researcher glancing at nine items in their panel has no frame of reference for what "Found on OpenAlex" means or whether to act on it. The pane body correctly renders "Citation tracking for books is limited in OpenAlex." (line 351), but the header summary — which is the only text visible when the pane is collapsed — says something different and less useful.

The two messages should form a coherent pair. Consider "Book — limited coverage" for the header, mirroring the pane body's explanation. Or at minimum, align the header to echo the body: "Limited coverage on OpenAlex."

**2. The Citation Network dialog still guards on DOI-only at `src/modules/citationNetwork/dialog.ts:44–49`**

```
const doi = item.getField("DOI");
if (!doi || !doi.trim()) {
  Services.prompt.alert(null, "Citegeist",
    "This item has no DOI. Citation network requires a DOI.");
  return;
}
```

This is fine architecturally (the network browser genuinely needs a DOI for OpenAlex's `/works/doi:` endpoint), but the user-facing alert is now inconsistent with the marketing story v1.1.0 ships: the README and pane both highlight multi-identifier support, and a researcher who just learned Citegeist supports ISBNs will try to open the citation network for their book and get a DOI-only error. This is a craft inconsistency, not a code bug — but it erodes trust.

**Fix:** Soften the alert: "Citation network browsing requires a DOI. This item was found via ISBN — try it on a journal article, or search this title in the citation network of a citing work." Alternatively, surface this limitation proactively in the pane's book zero-state.

## 🟡 Consider

- **"J. H-Index" column label is abbreviated differently from every other label.** The nine columns use full or initialism labels: "Citations," "FWCI," "Percentile," "Citedness," "UTD24," "FT50," "ABDC '25," "AJG '24" — and then "J. H-Index." The abbreviation style is inconsistent (period after J, space before H-Index). Consider "Journal H-Index" at full width or "H-Index (Journal)" to match the tooltip-driven pattern of the other labels. `src/modules/citationColumn.ts:246`

- **"Citedness" column label is technically correct but researcher-unfamiliar.** OpenAlex calls the metric "2-year mean citedness" and the README explains it as "an open equivalent of the Journal Impact Factor." In the column header, just "Citedness" provides no frame of reference. A researcher in the life sciences will not recognize it. "Mean Citedness" or "2yr Citedness" (matching the code's internal key `COL_CITEDNESS = "citegeist-citedness-2yr"`) would be more self-explanatory. `src/modules/citationColumn.ts:231`

- **Book zero-suppression in pane renders the action buttons for a work that has no trackable citation history.** When a book has zero citations, the headline stat line is suppressed and replaced with "Citation tracking for books is limited in OpenAlex." — but the "View 0 citing works →" and "View references →" action buttons are still rendered below it (`src/modules/citationPane.ts:379–393`). The citing-works button will open the dialog and immediately hit the DOI guard or show an empty state. A researcher clicking it after reading "limited coverage" will expect either a disabled button or a better-scoped label ("View references →" is valid and useful; the citing-works button is the one to reconsider).

## ⚪ Delights missed

- **The trend line is a genuine UX craft move and deserves visual differentiation.** The rising/falling indicator (`↗ 45 citations in 2024 (+23%)`) is the right call — it replaces a sparkline with an actionable insight. But it renders in the same `cg-no-doi` muted grey as every other secondary string. The `↗`/`↘` arrows would land harder if the rising case used the sage green (`#8FAD9F`) and the falling case used the muted red. One CSS class swap per direction. `src/modules/citationPane.ts:418–430`

- **The "Default folder:" label in the network dialog footer is UI chrome that the researcher doesn't need to see.** The label sits left of the chip on every page load, using the sentence-case "folder" when every other reference in the product uses "collection." Consider hiding the label entirely and letting the folder chip be self-descriptive, or aligning terminology to "collection" throughout. `src/modules/citationNetwork/dialog.ts:280`
